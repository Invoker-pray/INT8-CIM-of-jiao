### Part 2 — Priority 3 / Architecture extensions (pick the strongest one)

I gave the project author three options earlier. **Evaluate all three rigorously, then recommend exactly one** as the primary post-C1 work, plus one as a stretch goal.

#### Option A — Hardware-fused MaxPool / BatchNorm

- The author's CIMModel currently runs Pool layers in Python on the PS, which means an extra read-out + im2col round-trip per Pool layer. After C3, MMIO is no longer ~100 ms but it is still nonzero per round-trip.
- Design a `cim_pool_unit.sv` that sits beside `cim_accel_core.sv` and reads the output buffer directly, computes 2×2 max with stride 2, writes back to input_buffer for the next layer — no PS round-trip.
- BatchNorm: argue whether folding BN into the existing requantize multiply-shift is sufficient (it is, for inference-only INT8 — show the math: `BN(x) = γ(x-μ)/σ + β` collapses into a per-channel scale + bias that merges with the existing requantize `mult/shift` and bias_sram). Decide whether anything new needs to be added to RTL or if it's purely a `model_zoo.py` quantization-script change.
- Specify the CSR additions, FSM additions in the accel core (or a new top-level scheduler that chains Conv → Pool → next Conv without PS intervention), and the testbench plan.

#### Option B — Output-sparsity / zero-skip exploitation

- Post-ReLU activations are typically 40-60% zero. Show measured sparsity for LeNet-5 and MLP layers using the existing `sw/logs/` per-layer dumps (the verify-per-layer infra already saves `y.hex` per layer — this gives real numbers, not estimates).
- Design a `nonzero_index_buffer` that, after each layer, scans the output buffer and writes a list of nonzero indices into a small BRAM. The next layer's `cim_accel_core` consumes this list and skips zero columns in its MAC inner loop.
- Address: control overhead vs compute saving cross-over (skip is only worth it when fraction of nonzero × compute > scan cost). Quantify for FC2 (128→10) and FC1 (784→128) of the MLP and for the FC layers of LeNet-5.
- Cite the relevant prior work in your reasoning (e.g. SCNN, EIE, Cambricon-X — give exact paper titles, not just author lists).

#### Option C — DSP48 INT8×2 SIMD packing (Xilinx XAPP1163)

- Each DSP48E1 in 25×18 mode can compute two parallel 8×8 multiplies whose results don't overflow into each other when one operand is shared. This effectively doubles MACs/DSP if you pack two output rows per DSP.
- Show the 25-bit input layout (`{w_a[7:0], 9'b0, w_b[7:0]}`) and prove the result lanes don't collide for our INT8 ranges (worst-case `127 × 127 × 16` = 2,064,384 = 21 bits, fits comfortably below the 9-bit guard).
- Re-derive the cim_tile structure: 16×16 = 256 MACs would now use 128 DSP48 instead of 256, freeing half for either parallelism (PAR_OB=2 for free) or other operators.
- This is **the most likely path to "more compute in the same fabric"** without changing BRAM, which matters because step 5 (KV260) is out of scope so we cannot get more DSPs by moving fabric.

For each option provide:
- A clear architectural sketch and signal list.
- Quantified expected gain (with explicit math, not hand-waving).
- Verification strategy.
- Risk + rollback.
