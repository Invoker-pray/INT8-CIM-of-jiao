\cabstract{
随着深度学习在边缘智能场景中的广泛部署，传统冯·诺依曼架构中处理器与存储器分离所带来的数据搬运瓶颈日益突出。存算一体（Computing-in-Memory, CIM）技术通过在存储阵列内部或近存储位置完成计算，从体系结构层面缓解"内存墙"问题。

本文设计并实现了一种面向边缘推理的INT8数字存算一体矩阵计算阵列架构，在PYNQ-Z2（Zynq-7020）FPGA平台上完成了综合、部署与板级验证。系统以16×16 CIM Tile为原子计算单元，通过乘法器时分复用将单Tile的DSP48占用由256降至约64，在220块DSP的资源预算内部署输出并行度PAR\_OB=2/3（周期级仿真扫描至16）。核心引擎通过多级流水线状态机实现权重加载、零点减法、MAC运算、偏置累加、ReLU激活、重量化与结果写回的完整推理数据通路。数据通路采用AXI4-Stream + DMA双向批量传输替代逐字MMIO，配合输入/输出双缓冲乒乓结构与OBUF→IBUF层融合机制，将LeNet-5端到端延迟从约3,194 ms/幅降至29.88 ms/幅（约107倍数据通路加速）。软件侧引入块对角权重打包（SQ-mapping），将稀疏小卷积核场景下的MVM调用次数削减90\%以上。控制路径方面，集成了PicoRV32 RISC-V软核，通过总线桥接与双端口BRAM实现纯PL侧自主推理，且CIM IP完全不需修改。

实验结果表明，在64 MHz签核频率（PAR\_OB=2，WNS +0.013 ns）下，MLP网络（784→128→10）端到端延迟为1.94 ms/幅（约513 fps），LeNet-5 CNN端到端延迟为29.88 ms/幅（约33.5 fps），分类准确率99\%--100\%，全部硬件输出与自研Python整数golden model逐元素bit-exact一致。在与ARM Cortex-A9纯软件（NumPy/BLAS）同口径对比中，CIM硬件计算核快约65--191倍，系统级端到端仍快约1.23--2.24倍。PicoRV32模式以更少的逻辑资源（FF减少约2136个）实现纯PL自主推理，全部logits与ARM路径bit-exact一致。
}

\eabstract{
With the widespread deployment of deep learning at the edge, the data movement bottleneck caused by the separation of processor and memory in traditional von Neumann architectures has become increasingly prominent. Computing-in-Memory (CIM) technology mitigates the ``memory wall'' problem by performing computation within or near memory arrays.

This thesis designs and implements an INT8 digital CIM matrix computation array architecture for edge inference, with synthesis, deployment, and board-level verification on the PYNQ-Z2 (Zynq-7020) FPGA platform. The system employs a 16$\times$16 CIM Tile as the atomic compute unit; multiplier time-multiplexing reduces the per-tile DSP48 count from 256 to approximately 64, enabling output parallelism PAR\_OB=2/3 within the 220-DSP budget (with a cycle-level simulation sweep up to 16). The core engine implements the complete inference datapath—weight loading, zero-point subtraction, MAC operation, bias accumulation, ReLU activation, requantization, and result writeback—through a multi-stage pipelined FSM. The data path adopts an AXI4-Stream + DMA bidirectional bulk-transfer architecture in place of per-word MMIO, combined with input/output double-buffer ping-pong and OBUF$\rightarrow$IBUF layer fusion, reducing the LeNet-5 end-to-end latency from approximately 3,194 ms/image to 29.88 ms/image (about a 107$\times$ data-path speedup). On the software side, block-diagonal weight packing (SQ-mapping) is introduced to cut the number of MVM invocations for sparse small-kernel convolutions by over 90\%. For the control path, a PicoRV32 RISC-V soft-core is integrated; with a bus bridge and dual-port BRAMs, the accelerator achieves autonomous inference in pure programmable logic with zero modification to the CIM IP.

Experimental results show that at the 64 MHz signoff frequency (PAR\_OB=2, WNS +0.013 ns), the MLP network (784$\rightarrow$128$\rightarrow$10) achieves 1.94 ms/image end-to-end (about 513 fps), while LeNet-5 achieves 29.88 ms/image (about 33.5 fps), with classification accuracy of 99\%--100\%. All hardware outputs are element-wise bit-exact with a custom Python integer golden model. In an apples-to-apples comparison against pure ARM Cortex-A9 software (NumPy/BLAS), the CIM hardware is about 65--191$\times$ faster at the compute-core level and remains 1.23--2.24$\times$ faster at the system level. The PicoRV32 mode achieves autonomous inference in pure programmable logic with fewer logic resources (approximately 2,136 fewer FFs than ARM mode), and all logits are bit-exact with the ARM path.
}

\section{选题背景}

近年来，以YOLO\cite{10}、SSD\cite{11}为代表的深度学习目标检测算法在智能安防、工业视觉、无人系统与车载感知等场景中被广泛采用，其共同特点是需要在较高精度下实现接近实时的推理能力，而网络内部又以卷积/矩阵乘加（MAC）为主导计算负载，导致整体计算量与数据访问量都十分庞大。YOLO将检测任务端到端地建模为单网络回归问题，SSD则是通过多尺度特征图上的卷积预测实现单阶段检测，这类方法在追求速度与精度的同时，也使得卷积算子成为系统性能与功耗优化的核心瓶颈之一。

进一步观察可以发现，无论是YOLO、SSD这类检测网络，还是更轻量的多层感知机（MLP）与LeNet-5等分类网络，其主干运算都可以归结为矩阵-向量乘法（Matrix-Vector Multiplication, MVM）：全连接层本身就是一次MVM，而卷积层在经过im2col变换之后同样可以转化为矩阵乘法的形式。这意味着，只要硬件能够高效地完成维度可配置的INT8 MVM运算，并在其后衔接偏置累加、激活函数与重量化等定点后处理，便可以覆盖绝大多数边缘推理网络的核心算力需求。这一观察是本文将设计目标聚焦于"面向任意层的通用INT8 MVM计算阵列"、而非针对某一特定网络定制专用电路的根本出发点；它使得同一套RTL能够通过运行时配置服务于不同规模、不同结构的网络，从而在工程上获得更好的复用性与可维护性。

然而，在传统冯·诺依曼架构中，处理器与存储器相互分离，权重与特征图等数据需要在存储层级与计算单元之间频繁搬运，导致数据搬运资源开销远比实际计算要大，系统性能逐渐受限于存储带宽与访问延迟，"内存墙（Memory Wall）"问题由此形成并长期存在（如图\ref{fig:vn_vs_cim}所示，图中左侧传统架构中计算单元与存储器分离，数据需在层级间频繁搬运，受"内存墙"制约）。Wulf与McKee早在1995年就指出处理器速度提升与存储器速度提升之间的差距会不断扩大，从而限制系统级性能提升\cite{1}。与此同时，能效维度的矛盾更加突出：在先进工艺条件下，数据搬运（尤其是跨层级、跨片外存储访问）相对于算术运算存在着显著更高的能量代价，使得"算得快"并不等价于"算得省"。Horowitz在ISSCC 2014的报告中系统讨论了计算系统的能耗拆分，强调需要从体系结构与数据流层面减少昂贵的数据访问\cite{2}。面向CNN的专用加速器研究也进一步表明：通过数据复用与数据流优化降低（尤其是片外）访存开销，是实现低功耗实时推理的关键路径之一\cite{3}。在边缘计算场景（比如电池供电、散热受限和实时性要求较高的场景）中，上述矛盾更加尖锐，传统"分离式计算-存储"模式难以同时满足吞吐与能效需求。图\ref{fig:energy_per_op}对比了典型算术运算与不同层级存储访问的单次操作能耗（对数坐标，45nm工艺典型值\cite{2}），可见片外DRAM访问的能耗比一次8-bit MAC运算高出约两个数量级，直观地解释了想要实现低功耗推理为何要同时降低数据搬运而非单纯提升算力。

存算一体（Computing-in-Memory, CIM）技术通过在存储阵列内部或近存储位置完成计算，将"数据搬运"转化为"原位计算"，从体系结构层面缓解冯·诺依曼瓶颈（如图\ref{fig:vn_vs_cim}所示，右侧为存算一体架构，将计算单元嵌入存储阵列内部，实现存储与计算的位置协同，显著减少数据搬运）。Nature Electronics的一些综述工作提出了覆盖不同存储器参与程度的CIM全谱系分类框架，并指出CIM可在逻辑运算与乘加运算等基本原语上实现能效提升\cite{4}。在诸多实现路径中，基于标准CMOS工艺的SRAM-CIM具有速度快、工艺成熟、可靠性高、与主流数字设计流程兼容等优点，是面向工程落地与系统集成的重要路线之一\cite{5}。Chuan-Jia Jhang团队系统综述了SRAM-CIM在AI边缘设备中的挑战与发展趋势\cite{30}，Si Xin与Dong Fangyuan团队也从设计挑战与方法学角度对SRAM-CIM进行了全面回顾\cite{31}\cite{34}。目前来讲，CIM技术被认为是突破内存墙、提升能效与吞吐率的重要方向。

从应用角度看，为在精度、硬件复杂度与系统吞吐之间取得平衡，INT8量化推理已成为工业界与学术界常用的落地方案之一。Jacob等人提出的整数算术推理量化方法表明，通过合理的量化尺度与训练以及恰当的校准策略，可在保持精度的前提下实现高效的整数运算推理流程\cite{6}。从硬件角度看，已有研究在SRAM-CIM宏单元中展示了面向多比特MAC的实现范式，如8b MAC的SRAM CIM宏以及支持Signed-INT8的SRAM CIM宏等工作，说明"多比特、可用于实际推理精度需求"的SRAM-CIM在工程上具有可行性\cite{8}\cite{32}。另外，在硬件加速方面，FPGA因其可重构性、低延迟与较高的能效比，已成为神经网络推理加速的重要平台\cite{meng2025}，已有工作在此基础上探索了注意力机制等特定算子的FPGA硬件加速\cite{feng2024}。

基于上述背景，本课题面向边缘智能推理的能效与实时性需求，在标准CMOS SRAM工艺假设下，设计了一种支持INT8数据类型的数字存算一体矩阵计算阵列架构（设计空间定位与两种实现极端的分析见§\ref{sec:two_extremes}）：以字级近存储方式组织权重存储与乘加及部分积生成（权重驻留BRAM、乘加由DSP48完成），结合合理的阵列并行组织、累加与数据通路调度，降低数据搬运开销并提升并行吞吐；通过Synopsys公司的VCS工具完成仿真测试，在AMD公司的Vivado(2024.2版本)完成综合和布局布线，并在FPGA平台完成实际综合部署与功能验证，形成了可复现的验证平台与完整代码交付。

\section{国内外现状及发展趋势}

存算一体研究近年来快速发展，已形成覆盖器件/电路/架构/算法的多层次技术体系。相关工作从实现介质（SRAM、DRAM、RRAM/PCM等）与计算形态（模拟/数字、阵列内/近存储）对CIM进行系统分类，指出其共同目标是减少访存与数据移动开销、利用阵列并行来提升吞吐\cite{4}\cite{5}\cite{12}\cite{14}\cite{30}\cite{31}\cite{34}\cite{r12}。近五年来，全数字SRAM-CIM在能效与精度上持续突破，相关宏单元与处理器工作密集涌现\cite{r1}\cite{r6}\cite{r8}\cite{r9}，成为面向边缘AI的重要研究热点。图\ref{fig:cim_taxonomy}按存储介质（SRAM/DRAM/RRAM/PCM等）与计算形态（模拟/数字、阵列内/近存储）给出了CIM技术的谱系分类，并标注了各分支的代表性工作；本文的设计落在"SRAM-CIM → 数字SRAM-CIM"这一分支（图中红框高亮），与ReDCIM、TranCIM、MulTCIM等数字CIM工作处于同一技术路线。


在国际研究中，基于新型非易失存储器（如RRAM）交叉阵列的模拟CIM方案曾是重要热点，典型代表包括ISAAC\cite{17}和PRIME\cite{18}等工作。PUMA等可编程memristor加速器进一步探索了可编程的推理加速路径\cite{19}。近年来，基于ReRAM、STT-MRAM等新型器件的多比特CIM宏单元也在持续推进，进一步提升了边缘AI场景下的能效\cite{r11}\cite{r10}。但模拟CIM往往需要ADC/DAC等外围电路并面临器件非理想性、噪声等的影响，其精度与可校准性面临不小的挑战\cite{4}\cite{15}。面向通用存储体系的近存储计算也形成了多条路线，如Ambit\cite{21}、Neural Cache\cite{20}、FloatPIM\cite{22}等。

在电路与工程实现层面，基于标准CMOS工艺的SRAM-CIM因其速度、成熟度与数字设计流程兼容性而成为当前研究与产业化的重要路线\cite{5}\cite{12}\cite{14}。已有多项芯片级宏单元工作给出了可参考的实现范式，比如在28nm工艺下实现的64Kb 6T SRAM-CIM宏单元\cite{32}，以及支持Signed-INT8的ADC-less SRAM-CIM宏单元\cite{8}；近五年更涌现出面向更高精度与能效的多种宏单元设计，如POSIT数制CIM宏\cite{r2}、精度可重构数字CIM宏\cite{r3}、Booth编码近似数字CIM宏\cite{r5}，以及面向低功耗边缘的8T SRAM-CIM\cite{r16}与电流镜式SRAM-CIM宏\cite{r17}等。研究趋势正从"单个CIM宏单元验证"走向"可重构/可编程的数字CIM处理器与系统级加速器"。涂锋斌老师团队提出了支持BF16与INT8的可重构数字CIM处理器\cite{24}，其后续ReDCIM工作给出了统一FP/INT的数字CIM设计\cite{25}。同期也出现面向Transformer/注意力工作负载的数字CIM加速研究，如TranCIM\cite{26}、MulTCIM\cite{27}、面向注意力的近似数字CIM\cite{r4}等，以及支持浮点推理/训练的稀疏-数字CIM处理器\cite{r6}与新型动态逻辑CIM宏\cite{r7}。

在算法侧，INT8量化推理已成为常用的落地路径。Jacob等工作提出的整数算术推理量化方法提供了硬件采用INT8数据通路的算法依据\cite{6}。PACT\cite{28}与LSQ\cite{29}等研究进一步从训练与量化参数学习角度改善量化精度。在能效方面，Dong Qing等在7nm FinFET工艺下实现了351 TOPS/W的SRAM-CIM宏单元\cite{33}，展现了先进工艺下CIM的巨大能效潜力。

上述芯片级宏单元工作大多以ASIC流片或宏单元级别的能效、面积指标对比为主，而对"如何把一个CIM计算阵列接入完整的SoC数据通路、在真实硬件平台上端到端跑通一个神经网络、并保证逐元素数值正确"这一系统级问题着墨相对较少。在以FPGA为载体的神经网络推理加速方面，近年来也已有大量工作通过脉动阵列、INT8/低比特量化、数据复用与层间流水等手段提升端到端吞吐与能效\cite{r14}\cite{r15}，但其多以独立加速器为主，与CIM计算范式的结合仍有探索空间。对于以FPGA为载体的原型验证而言，除了计算核心本身的设计之外，权重/输入/输出的批量搬运效率、层与层之间的数据调度方式、以及软硬件数值一致性的可验证性，往往才是真正决定端到端吞吐率与结果可信度的关键环节——一个MVM算得再快，如果数据搬运成为瓶颈、或者无法证明硬件输出与软件参考逐位一致，其工程价值都会大打折扣。本课题正是面向这一相对薄弱的环节，强调在资源受限的小型FPGA上建立一套完整的工程验证闭环，而非单纯追求某一项孤立的峰值指标。

综合来看，国内外CIM研究呈现如下趋势：技术路线从"模拟/新型器件主导"扩展为"模拟与全数字并行发展"；算子与精度从低比特走向多比特MAC（尤其是Signed-INT8）；应用从CNN拓展到Transformer/多模态；验证方式越来越需要体系结构层面的可复现评估（包括RTL/FPGA原型验证、资源/频率/吞吐/延迟评估等）\cite{12}\cite{16}。本课题正是在此背景下，以数字SRAM-CIM阵列架构为核心，围绕数据通路、控制调度与FPGA验证建立了工程闭环。

\section{本文主要工作与贡献}

本文完成了以下工作：

\section{数字SRAM-CIM的两种实现极端与本设计定位}
\label{sec:two_extremes}

在进入具体的模块设计之前，有必要先界定本课题在整个数字SRAM-CIM设计空间中的位置。现有数字SRAM-CIM的实现方式虽然种类繁多，但若按"单个存储位参与计算的粒度"这一维度来观察，可以抽象出两种典型的实现极端：一端是比特级bit-serial popcount架构，另一端是字级行为建模MAC架构。理解这两端的权衡，有助于解释为什么本课题在PYNQ-Z2资源约束下选择了后者。

\subsection{比特级 bit-serial popcount 架构}

这种架构把存储单元直接用作一位乘法器。该架构的典型形态是：SRAM cell只存储1位权重$w_i$，输入$x_i$的一位以WL广播进入阵列，BL/BLB上得到$x_i \cdot w_i$（实质是AND运算），再由阵列底部的加法器树对一整列做popcount累加得到一位×一位的点积。为支持INT8精度，输入的8个bit-plane需要在时间维度上顺次送入阵列，每次popcount结果按对应位权做移位累加，最终得到INT8$\times$INT8的MAC结果。

这种架构最贴近"存算一体"的本义：计算真正发生在存储阵列内部，每个bit-cell同时承担存储和计算角色，天然具备高并行度（一次popcount可覆盖数百行）、低数据搬运（输入只沿WL广播）和小面积（单cell仅含几颗晶体管）的优势。典型的阵列规模可达128行$\times$128列的1-bit cell，每8列打包为一个INT8权重通道，通过8-状态的bit-plane有限状态机实现INT8 MAC。

这一路线的代价也很直接：（1）一个INT8 MAC需要至少8个时钟周期完成，时序延迟被拉长为字级的$8\times$；（2）bit-serial流水对FPGA综合器并不友好，FPGA上很难推断出紧凑的BRAM+DSP结构，通常只能走LUT+寄存器，资源利用率反而偏低；（3）精度后处理（bias累加、ReLU、重量化）很难直接融入bit-serial流水，只能把INT32部分积搬回CPU处理。因此bit-serial架构在ASIC下确实有真正的面积与能效优势，但在面向FPGA原型验证和端到端闭环的场景下，性价比并不突出。

\subsection{字级行为建模MAC架构}

另一种架构则是把CIM阵列抽象为字级的行为模型：存储子系统按INT8/INT32字存取，计算单元一个周期内直接完成INT8$\times$INT8乘法与列方向累加，不再展开到bit-plane时间维度。在FPGA平台上，这种抽象可以被综合器自动映射到DSP48E1原语，每个DSP48能在一个时钟周期内完成一次有符号$18\times25$乘法与48位累加，对INT8 MAC是完全够用的。本课题的\texttt{cim\_tile.sv}即采用这一思路：16$\times$16的纯组合阵列在一拍内完成256个INT8 MAC，通过列方向的链式累加获得16个INT32部分和。

该路线带来的直接收益是：（1）单MAC延迟仅1拍，数据通路时序简洁，易在PYNQ-Z2上收敛至较高频率；（2）与传统数字综合流程完全兼容，可利用Vivado的DSP推断与BRAM推断自动优化；（3）因为计算单元是字级而非比特级，可以在MVM之后的同一条流水线中无缝嵌入bias累加、ReLU与multiply-shift重量化模块，做到INT8输入→INT8输出的端到端硬件闭环，PS仅在层与层之间介入；（4）单一FSM引擎通过CSR配置即可服务任意层，避免了为每一层设计专用硬件的冗余。

代价同样明显：（1）存储单元退化为常规BRAM，"存算一体"的物理含义被弱化，更接近近存储计算（near-memory computing）而非真正的in-memory computing；（2）DSP48数量成为硬性上限，PYNQ-Z2上只有220块DSP，若每个Tile行将16列乘法器全部组合实例化，单Tile即占满DSP而无法部署多份。为此本设计采用乘法器时分复用（详见后文"CIM Tile设计"一节）将单Tile的DSP占用压缩至约64块，从而在220块DSP的预算内部署多份Tile（板级PAR\_OB=2/3）。因此该路线本质是"牺牲CIM的物理严格性，换取FPGA平台下的工程可落地性与端到端闭环能力"。

\section{整体架构}

本设计采用"输入缓冲—CIM阵列—累加与量化—输出缓冲"的流水式组织，核心引擎以有限状态机（FSM）驱动完整的推理数据通路。系统顶层架构如图\ref{fig:top_arch}所示：ARM PS（或PicoRV32软核）通过AXI Interconnect以AXI4-Lite配置CSR、以AXI4-Stream（MM2S/S2MM）经AXI DMA与CIM IP批量交换数据；CIM IP内部由接口层（AXI4-Lite Slave、AXIS Sink/Source）、计算层（cim\_accel\_core、cim\_tile、psum\_accum、requant）与存储层（weight\_sram、bias\_sram、input/output\_buffer）三部分构成，图中箭头按数据类型着色：权重（红）、输入（蓝）、psum（绿）、INT8结果（黄）。系统由以下关键模块组成：
\begin{itemize}
	\item cim\_accel\_core：核心加速器引擎，包含多状态FSM与多级流水线，负责计算调度与性能计数；
	\item cim\_tile：16×16 CIM原子计算单元，采用乘法器时分复用（每行实例化\newline TILE\_COLS/TILE\_SPLIT\_FACTOR=4个物理乘法器，由FSM分4拍覆盖16列），等效完成256个INT8 MAC运算；
	\item psum\_accum：部分和累加器，带优先级清零机制；
	\item weight\_sram：16-bank权重SRAM，支持AXI DMA写入与tile-packed读取；
	\item bias\_sram：偏置SRAM，按输出神经元索引寻址；
	\item input\_buffer：输入缓冲，双bank乒乓结构，支持AXI写入与零点减法；
	\item output\_buffer：输出缓冲，双bank乒乓结构，含argmax逻辑；
	\item cim\_axi\_stream\_sink：AXI4-Stream slave，接收DMA批量数据并路由至各存储模块；
	\item cim\_axi\_stream\_source：AXI4-Stream master，将输出缓冲结果通过DMA批量回传；
	\item cim\_axi\_lite\_slave：AXI4-Lite接口，提供CSR配置、存储窗口访问及stream/legacy路径MUX。
\end{itemize}

所有可配置参数集中管理于\texttt{cim\_pkg.sv}文件中，包括Tile尺寸（TILE\_ROWS=16, TILE\_COLS=16）、并行度（PAR\_OB）、MAC拆分因子（TILE\_SPLIT\_FACTOR）、数据位宽（INT8输入/权重、INT32偏置/累加器）、最大层维度（MAX\_IN\_DIM=1536, MAX\_OUT\_DIM=256）、CSR地址映射与FSM状态编码。保证项目不存在magic number，所有RTL模块通过\texttt{import cim\_pkg::*}引用参数。

加速器的工作流程为：PS（或PicoRV32）通过AXI4-Stream DMA接口批量写入权重、偏置与输入数据，通过AXI4-Lite接口配置层参数（IN\_DIM, OUT\_DIM, N\_IB, N\_OB, requant\_mult, requant\_shift, act\_mode, input\_zp），然后触发start信号。核心引擎按output-block并行、input-block迭代的顺序完成MVM运算，每轮计算后进行偏置累加、激活与重量化，最终将INT8结果写入输出缓冲。完成后置位done信号并可触发IRQ。对于连续FC层，层融合机制可将OBUF结果通过内部硬件路径直接拷贝至IBUF，避免DMA往返。

上述"接口层—计算层—存储层"的三层划分把"会随平台/数据通路演进而改变的部分"与"承载核心算力、应当保持稳定的部分"解耦：接口层（AXI4-Lite Slave、AXIS Sink/Source）负责与外部总线和DMA交互，是后续从ARM切换到PicoRV32、或从逐字MMIO切换到AXI4-Stream时唯一需要改动的区域；计算层（cim\_accel\_core、cim\_tile、psum\_accum及重量化逻辑）封装了与控制方式无关的纯计算流水；存储层则以统一的整行/整字写接口对上屏蔽了bank拆分等实现细节。正因如此，第3章用RISC-V软核替换ARM控制时，计算层与存储层可以做到"零RTL改动"。在计算调度上，核心引擎采用"外层按output-block、内层按input-block"的二重循环：外层每次处理由\texttt{PAR\_OB}个Tile行并行覆盖的一组输出神经元，内层沿输入维度逐个input-block地累加部分和；当某组输出的所有input-block累加完毕后，再统一对这组输出做偏置累加、激活与重量化并写回OBUF，然后进入下一组output-block。这种组织方式使得偏置累加、激活、重量化等"每个输出神经元只做一次"的后处理被自然地从内层MAC循环中分离出来，避免了重复计算。

\section{CIM Tile设计}

CIM Tile是本设计的原子计算单元，实现以下运算：

\begin{equation}
	\text{psum}[r] = \sum_{c=0}^{15} \text{x\_eff}[c] \times w[r][c], \quad r = 0, 1, \ldots, 15
\end{equation}

其中$\text{x\_eff}$为经过零点减法后的10位\emph{有符号}有效输入（位宽\texttt{X\_EFF\_W=10}，取值区间$[-512,511]$），$w$为Signed-INT8权重，输出为32位有符号部分和。本文所有MNIST/LeNet模型输入侧取$\text{input\_zp}=0$，UINT8像素$[0,255]$经零扩展后即作为有效输入$\text{x\_eff}\in[0,255]$（恰为有符号区间中的非负子集）；$\text{input\_zp}$为按层可配置的\emph{有符号}CSR量，当其取非零值时$\text{x\_eff}=x-\text{input\_zp}$可为负数，故采用有符号10位表示以正确容纳带非零零点的其他量化约定。

Tile采用纯组合逻辑实现，通过generate构造16行×16列的乘加阵列。每行独立计算一个输出神经元的部分和，列方向采用链式累加：

\begin{equation}
	\text{row\_acc}[c+1] = \text{row\_acc}[c] + \text{signed\_extend}(\text{x\_eff}[c]) \times w[r][c]
\end{equation}

在\texttt{TILE\_MAC\_REUSE=1}的默认配置下，每行并不组合实例化16个乘法器，而是仅实例化\texttt{TILE\_COLS/TILE\_SPLIT\_FACTOR}=4个物理乘法器，由核心引擎FSM在4个时钟周期内依次喂入第$q$组（$q=0,1,2,3$）的4列\texttt{x\_eff}与权重，从而以时分复用方式覆盖全部16列、等效完成$16\times16=256$次MAC。采用这种"少量乘法器+时分复用"设计的核心动机，是把单Tile的DSP48占用从256降至约64，使得在仅有220块DSP48的PYNQ-Z2上仍可部署多份Tile（\texttt{PAR\_OB}=2/3）实现输出并行。其内部结构如图\ref{fig:cim_tile}所示：有效输入$\text{x\_eff}$沿行方向广播，权重沿列方向由对应bank送入，每行的4个乘法器输出经寄存器打拍后通过短链式加法得到该组部分积，4组累加后得到一个INT32部分和。

对零点的处理如下。原始像素是$[0,255]$的无符号UINT8，而权重是有符号INT8；若把UINT8直接按有符号解释再与权重相乘，会在像素值$\ge 128$时产生符号歧义。为此，硬件在从输入缓冲读出时先对$x\_tile$做零扩展（对应RTL中的\texttt{\$signed(\{1'b0, x\_tile\})}），再减去CSR配置的有符号零点得到有效输入，即\,\texttt{x\_eff = \$signed(\{1'b0, x\_tile\}) - input\_zp}\,，其结果按\emph{有符号}10位（\texttt{X\_EFF\_W=10}，区间$[-512,511]$）表示并在越界时做饱和。采用有符号位宽是数值语义正确的前提。对本文涉及的全部MLP与LeNet-5模型，输入侧均取$\text{input\_zp}=0$、权重采用对称量化（zp=0），此时$\text{x\_eff}$即像素值本身、范围$[0,255]$（落在有符号区间的非负子集内），不引入任何额外偏移项，乘法为"有符号10位$\times$有符号8位"，golden model严格复现同一有符号零点约定，从而与硬件输出逐位一致。进一步地，\texttt{input\_zp}被保留为按层可配置的有符号CSR以兼容带非零输入零点的量化约定；在标准整数量化中，非零输入零点$z$所引入的交叉项$z\cdot\sum_{c} w[r][c]$会在离线标定时按输出通道折入INT32偏置，因此无论零点取何值，硬件都只需执行同一条"零点减法$\to$MVM$\to$加偏置"流水即可保持数值正确。在位宽方面，单个有符号$10\text{-bit}\times 8\text{-bit}$乘积的幅值不超过$512\times128=2^{16}$（约17位幅值），沿16列累加后理论上限约$2^{20}$（约21位幅值），因此用32位有符号部分和寄存器存放绝不会溢出，留有充足裕量。在流水实现上，Tile对每行乘法器的输出先打一拍寄存器再做链式累加（\texttt{C4\_MUL\_PIPE}），把原本"DSP乘法$\to$进位链累加"的长组合路径切分为"寄存$\to$DSP乘法$\to$寄存"与"寄存$\to$CARRY4累加$\to$寄存"两段，从而显著缩短关键路径、为更高时钟频率与双缓冲重叠预留时序裕量；与此同时，规则的乘加结构仍可被综合器识别并映射到FPGA内置的DSP48E1硬核上——每个DSP48E1可在一拍内完成一次有符号$18\times25$乘法与48位累加，对10位输入、8位权重的INT8 MAC而言绰绰有余，相比用查找表搭建乘法器，资源与时序都更优。

为支持更高时钟频率，Tile内部的16列MAC可通过编译期参数\texttt{TILE\_SPLIT\_FACTOR}拆分为多组顺序执行。当\texttt{TILE\_SPLIT\_FACTOR}=4时，16列被分为4组（每4列一组），分4个时钟周期计算：

\begin{equation}
	\text{psum}[r] = \sum_{q=0}^{3} \sum_{c=4q}^{4q+3} \text{x\_eff}[c] \times w[r][c], \quad r = 0, \ldots, 15
\end{equation}

每组仅含4个MAC + 单个CARRY4进位链，再叠加乘积寄存器（\texttt{C4\_MUL\_PIPE}）将"DSP乘法$\to$进位链累加"进一步切分为两段流水，单段组合路径大幅缩短。FSM相应地从单一\texttt{ST\_MAC}状态扩充为每组三个子状态——\texttt{ST\_MAC\_Q*\_L}（锁存第$q$组的\texttt{x\_eff}/权重切片）$\to$\texttt{ST\_MAC\_Q*\_M}（寄存乘积）$\to$累加，四组（Q0$\sim$Q3）顺序执行，因此每个input-block迭代的MAC段由原来的约1拍扩展为若干拍。这一拆分与流水化策略的代价是MAC段周期数增加，但其价值在于把关键路径切短，使设计在PYNQ-Z2上于64 MHz完成时序收敛（75/80 MHz为功能性超频，WNS为负但常温功能正确），并为非MAC流水级（bias/ReLU/重量化）与双缓冲乒乓重叠提供时序裕量。

本版本默认开启\texttt{TILE\_MAC\_REUSE=1}，采用"乘法器时分复用"机制：每个Tile行只实例化\texttt{TILE\_COLS/TILE\_SPLIT\_FACTOR}=4个物理乘法器，由FSM分4拍轮流处理16列的四个切片，因此单Tile实际占用的乘法器约为$16\times4=64$个（而非$16\times16=256$个），"产出一次完整256-MAC结果需4拍"对应的是物理乘法器被复用4次。正是这一压缩使得在仅有220块DSP48E1的PYNQ-Z2上可以同时部署2$\sim$3份Tile：\texttt{PAR\_OB}=2的MAC部分约占128块DSP（连同重量化等逻辑实测共134块、约61\%），\texttt{PAR\_OB}=3的MAC部分约占192块（实测共198块、约90\%），DSP占用不再是早期单Tile全实例化时的100\%，从而把"用满DSP却只有一份Tile"的低效利用，转化为"在DSP预算内换取真实的输出并行度"。综合后各配置的DSP/BRAM/LUT精确占用见表\ref{tab:util_final}。

\section{存储子系统}

\subsection{权重SRAM的BRAM推断问题与bank拆分方案}

权重SRAM的设计是本项目的关键工程挑战之一。初始设计中，weight SRAM为一整块2048位宽（$\text{TILE\_ROWS} \times \text{TILE\_COLS} \times 8 = 2048$位）的存储器。然而，Vivado对这种超宽memory推断BRAM本身就困难；更关键的是，如果用bit-select方式做32位chunk部分写入，Vivado会直接放弃BRAM推断，退化为纯寄存器，导致资源爆炸。

解决方案是将weight SRAM拆分为16个独立的bank，每个bank仅$\text{ROW\_W} = \text{TILE\_COLS} \times \text{WEIGHT\_W} = 128$位宽，对应Tile的一行权重。16个bank通过generate实例化，每个bank独立推断为BRAM：

\begin{equation}
	\text{bank\_mem}[r][\text{tile\_idx}] = \{w[r][15], w[r][14], \ldots, w[r][0]\}, \quad r = 0, \ldots, 15
\end{equation}

写入侧使用staging register将4个32位AXI写入累积为完整的128位行数据，然后通过单周期整字写入送入对应bank，确保了BRAM推断所需的"纯整字写入"条件。该bank拆分前后的对比如图\ref{fig:weight_bank}所示。图中上半部分表示原始2048位单块在32位部分写入下无法被Vivado推断为BRAM，退化为FF/LUT；下半部分表示的是拆分为16个独立128位bank，配合staging register将4个32位AXI beat组装为128位整行写入，每个bank均成功推断为BRAM。

这一"先攒齐128位、再整行写入"的read-modify-write式通路，本质上是在32位的AXI接口与128位的BRAM整行写入之间架设了一个宽度转换桥：上层软件仍按32位字（chunk）逐个送权重，硬件在staging寄存器里把连续4个chunk拼成一整行，只在第4个chunk到达时拉高一拍写使能、把整行一次性写进BRAM。这样既保持了对软件友好的32位写接口（与既有的\texttt{weight\_to\_chunks}打包格式兼容），又满足了综合器对BRAM"整字写、不做位选择部分写"的苛刻要求。BRAM推断是本设计资源可行性的关键前提：权重存储的最坏容量约为$16\times128\times1536/8\approx 384\text{ KB}$，若退化为寄存器实现，所需的触发器数量将远超xc7z020全片10万余个FF的规模，整个工程将无法满足资源约束；而一旦正确推断为BRAM，这部分存储几乎不占用LUT/FF，仅消耗Block RAM资源。因此这一拆分既是功能正确性的保证，也是资源可行性的前提。

\subsection{偏置SRAM与输入/输出缓冲}

偏置SRAM按输出神经元索引寻址，每个条目32位（INT32），深度为MAX\_OUT\_DIM=256。输入缓冲存储tile-packed的输入数据，每条目$\text{TILE\_COLS} \times \text{INPUT\_W} = 128$位。输入缓冲在读取时自动完成零点减法：

\begin{equation}
	\text{x\_eff}[c] = \text{input}[c] - \text{input\_zp}
\end{equation}

其中input\_zp通过CSR按层配置（有符号量）；本文所有模型输入侧取input\_zp=0，即x\_eff为UINT8像素值本身（落在有符号10位区间$[-512,511]$的非负子集$[0,255]$内）。当input\_zp取非零值时，$x-\text{input\_zp}$可能为负，故x\_eff以有符号10位表示，为非零零点的量化约定预留正确的数值语义与余量。输出缓冲存储INT8推理结果，并内置argmax比较逻辑，在末层推理完成后自动给出预测类别。

\subsection{IBUF/OBUF双缓冲乒乓设计}

\section{量化与重定标}

本设计的量化流程遵循标准INT8整数推理规范\cite{6}，确保与软件golden model的bit-exact一致性。完整的量化链路为：

\begin{equation}
	\text{psum} = \sum_c \text{x\_eff}[c] \times w[r][c] + \text{bias}[r]
\end{equation}

\begin{equation}
	\text{prod} = \text{psum} \times \text{requant\_mult}
\end{equation}

\begin{equation}
	\text{shifted} = \left\lfloor \frac{\text{prod} + 2^{\text{requant\_shift}-1}}{2^{\text{requant\_shift}}} \right\rfloor
\end{equation}

\begin{equation}
	\text{output} = \text{clamp}(\text{shifted}, -128, 127)
\end{equation}

其中requant\_mult和requant\_shift为每层可配置的量化参数，通过CSR在运行时设置。重定标过程中使用64位中间乘积以避免溢出，移位时采用带舍入的算术右移（round-half-up），最后clamp到INT8范围$[-128, 127]$。

从原理上看，这条重定标链路是浮点缩放因子的定点化近似。在标准INT8量化推理中，一层累加得到的INT32结果需要乘以一个介于0与1之间的实数缩放因子$M=S_xS_w/S_y$（由输入、权重、输出三者的量化尺度决定）才能回到下一层的INT8动态范围。硬件无法直接做浮点乘法，于是把$M$近似为$M\approx \text{requant\_mult}/2^{\text{requant\_shift}}$这样一个"定点乘数加右移"的组合：先用一个32位整数乘数与INT32部分和相乘，再算术右移指定位数。式中的$+2^{\text{requant\_shift}-1}$是四舍五入项，等价于在右移前加上半个最低有效位，使移位结果做到"就近取整"（round-half-up）而非简单截断，这一点对于把硬件输出与软件参考模型对齐到逐位一致至关重要——若硬件用截断、软件用四舍五入，二者在大量边界样本上就会产生$\pm1$的偏差。关于位宽，INT32部分和与32位乘数相乘最坏可达约$63$位，因此中间乘积必须用64位容器承载，否则会溢出；右移并截断回INT8之前再做一次$[-128,127]$的饱和clamp，防止个别超界值回绕。\texttt{cim\_pkg.sv}中以一个\texttt{requantize()}纯函数精确描述了上述运算（64位乘积、带舍入右移、饱和），仿真testbench、综合RTL与Python golden model三者复用同一套定义，从源头上保证了三层数值的一致性。

激活函数支持三种模式：无激活（ACT\_NONE）、ReLU（ACT\_RELU，将负值归零）、范围截断（ACT\_CLAMP）。模式通过CSR的ACT\_MODE寄存器配置。这三种模式已足以覆盖本文涉及的MLP与LeNet-5网络：隐藏层使用ReLU、输出层使用无激活（直接给出logits供argmax），而ACT\_CLAMP则为将来需要硬截断的层预留。完整的量化重定标数据通路及各级位宽与对应FSM状态如图\ref{fig:requant}所示。下图展示了从psum累加经加偏置、ReLU/clamp、乘requant\_mult（升至64位）、加舍入项、算术右移、clamp至INT8，逐级标注位宽（32$\to$32$\to$32$\to$64$\to$64$\to$32$\to$8）与对应FSM状态。

\section{AXI4-Stream DMA数据通路}

系统通过AXI4-Stream + Xilinx axi\_dma IP实现双向批量数据传输，替代传统AXI4-Lite逐字MMIO。

之所以要做这一重构，是因为在早期以AXI4-Lite逐字MMIO搬运数据的版本中，数据通路而非计算本身成为了端到端性能的绝对瓶颈。逐字MMIO的根本问题在于：从PYNQ的Python侧看，一次\texttt{mmio.write()}是一次系统调用（经由Python运行时、libc、\texttt{/dev/mem}映射直至产生一笔AXI事务），单次软件开销约数微秒，而真正的AXI4-Lite单事务硬件耗时仅约数十纳秒——也就是说，软件循环把每笔传输的有效成本放大了数十倍。对于权重数据量较大的网络，这一放大尤其致命：LeNet-5单图打包后的权重就有约数万个32位字，若全部用Python循环逐字写入，仅权重加载一项就需要上百毫秒，从而使逐字MMIO配置下LeNet-5的端到端延迟高达每张数千毫秒，其中绝大部分时间都耗在了纯软件循环上而非硬件计算。修复思路是按数据特性分流：对量小、对延迟敏感的CSR配置仍保留AXI4-Lite（适合\texttt{mmio.write}逐个写）；对量大、可批量的权重/输入/输出数据，改走AXI4-Stream + \texttt{axi\_dma}，由PYNQ的\texttt{pynq.lib.dma}一行调用即可发起一次连续DMA，把数千个字一次性从DDR经HP高性能端口搬入CIM，彻底摊薄每字的软件开销。


\subsection{MM2S写入通路}

权重、输入与偏置的加载走DDR→\allowbreak S\_AXI\_HP0→\allowbreak axi\_dma→\allowbreak M\_AXIS\_MM2S→\allowbreak cim\_axi\_stream\_sink的批量stream路径。\texttt{cim\_axi\_stream\_sink}内部实现4-beat到128-bit行装配：每4个32-bit AXI4-Stream beat组成一个完整的128-bit行写入，根据CSR配置的dest字段（0=weight, 1=input, 2=bias）路由至对应存储模块。CSR控制仍保留原AXI4-Lite接口，CIM计算核心（\texttt{cim\_tile}、\texttt{psum\_accum}、\texttt{cim\_accel\_core}）完全不动。

关键设计决策包括：（1）AXIS数据宽度=32-bit，与现有weight\_to\_chunks软件packing兼容，DMA内部利用Data Realignment Engine做64→32dwidth转换；（2）dual-path共存——在\texttt{cim\_axi\_lite\_slave}内部通过CSR\_CTRL[3]对写入路径做MUX，legacy MMIO路径保留用于A/B对照与回归隔离；（3）cfg\_start通过写CSR\_STREAM\_LEN隐式触发，原子化操作省去额外的CSR写开销；（4）s\_axis\_tready恒为1——BRAM写为单周期完成，sink内部无需排队，反压沿由DMA侧向PS端反向传播。该sink将4个32位beat装配为128位行并按dest字段路由的过程如图\ref{fig:axis_sink}所示。


\begin{figure}[H]
	\centering

从微结构上看，\texttt{cim\_axi\_stream\_sink}本质是一个由\texttt{cfg\_dest}选择目的地、按beat计数驱动的小型状态机。它内部维护一个128位移位寄存器：每当一个32位beat有效（\texttt{tvalid \&\& tready}）时，就把新数据移入寄存器高位、原有数据整体右移一个字；当累计到第4个beat时，4个32位字恰好拼成一整行128位，于是拉高一拍对应存储模块的整行写使能，并据当前行计数、tile计数给出写地址，随后行计数自增，写满16行即tile计数自增。状态机同时用\texttt{beat\_count}与传入的\texttt{cfg\_len}、\texttt{tlast}做交叉校验：若在到达预期beat数之前就收到\texttt{tlast}则报underflow、若超过预期仍未收到则报overflow，这些状态经CSR回报给软件用于断言。该设计的特点是让\texttt{s\_axis\_tready}恒为高：由于BRAM写是单周期完成、sink内部不存在需要排队的情形，因此无需对DMA反压；真正的反压沿（当DDR或总线暂时供不上数据时）由DMA引擎一侧自然向PS方向回传，sink只需被动接收即可。这样既简化了握手逻辑，又不会因sink侧的人为停顿而拖慢传输。

\subsection{S2MM读回通路}

推理结果的读回走output\_buffer→\allowbreak cim\_axi\_stream\_source→\allowbreak M\_AXIS\_RESULT→\allowbreak axi\_dma/S\_AXIS\_S2MM→\allowbreak S\_AXI\_HP1→\allowbreak DDR的批量stream路径。\newline\texttt{cim\_axi\_stream\_source}采用5状态FSM（IDLE→WAIT→WARMUP→READ→SEND）处理BRAM的2-cycle读延迟与非4对齐末字打包。每4个INT8结果打包为一个32-bit AXI4-Stream beat，末拍置tlast=1告知DMA结束接收。Python驱动侧采用direct register mode直接操作S2MM的DMACR/DA/LENGTH寄存器，并实现双缓冲交替机制避免缓冲区复用问题。

\section{Layer Fusion: 层间融合}

针对连续全连接层（如LeNet-5的FC3→FC4→FC5或MLP的FC1→FC2），标准数据流需要在层间执行完整的S2MM读回→DDR→MM2S重新加载循环。本设计在\texttt{cim\_axi\_lite\_slave}中实现了OBUF→IBUF内部硬件拷贝的层融合（Layer Fusion）机制。需要特别说明的是，本文所称"层融合"特指OBUF→IBUF的片上硬件直拷（层间数据在片内直接转发、不经DDR往返），与深度学习框架中将Conv+BN+ReLU等多个算子合并为单一计算核的"算子融合"（operator fusion）是不同的概念，二者不应混淆。

融合FSM包含4个状态（F\_IDLE→F\_WAIT\_MUX→F\_PACK→F\_WRITE），工作流程为：OBUF按字节读取INT8结果→在硬件内部重新打包为128-bit tile字（16 INT8/tile）→直接写入IBUF。通过F\_WAIT\_MUX状态处理OBUF读路径的流水线延迟（always\_comb MUX传播 + BRAM registered read共2周期），确保byte-to-tile装配的字节对齐正确性。

约束条件为融合元素数$n\_elements \leq \text{MAX\_OUT\_DIM} = 256$，单次融合耗时$n\_elements + 1 + \lceil n\_elements/16 \rceil \times 2$周期。以FC1→FC2（128元素）为例，融合延迟约1.5 $\mu$s，远小于等效的DMA S2MM+MM2S往返延迟（数十毫秒）。图\ref{fig:layer_fusion}（a）对比了有无层融合两种数据流的层间延迟，无融合时FC1$\to$FC2需经S2MM$\to$DDR$\to$MM2S往返（数十毫秒），融合时仅需OBUF$\to$IBUF硬件直拷（约1.5~$\mu$s）；（b）给出了融合FSM的状态转移（F\_IDLE$\to$F\_WAIT\_MUX$\to$F\_PACK$\to$F\_WRITE）。

为支持多层权重共存，系统中增设了CSR\_WEIGHT\_BASE和CSR\_BIAS\_BASE偏移寄存器，使不同FC层的权重/偏置可在SRAM中存储于不同offset。FC1权重存于tile 0..N1-1，FC2权重存于tile N1..N1+N2-1，batch推理时仅需切换base寄存器即可在不同层之间切换。Python驱动侧的\texttt{setup\_fc\_fused\_pair()}预加载双layer权重，\texttt{infer\_fc\_fused\_batch()}实现批融合推理，无需每张图像重新加载FC2权重。

	\centering
	\includegraphics[width=\textwidth]{pics/2-8.png}
	\caption{层融合（Layer Fusion）机制示意图。}
	\label{fig:layer_fusion}
\end{figure}


层融合之所以能带来数量级的层间延迟收益，根源在于它彻底绕开了"片上→DDR→片上"这条昂贵的往返路径。在没有融合的标准数据流中，上一层算完的INT8结果要先经S2MM通道DMA回DDR，再由下一层经MM2S通道DMA读回片上，这一来一回不仅要两次穿越HP端口，还要承担两次DMA描述符配置的软件开销，单次往返动辄数十毫秒；而融合路径让上一层的OBUF结果直接在片内通过硬件通路重打包后写入下一层的IBUF，全程不出片、不经DDR、不需软件介入，单次仅约$1.5\,\mu\text{s}$。这里需要处理的一个工程细节是数据排布的转换：OBUF以字节（INT8）为粒度存放结果，而IBUF要求以128位tile字（每字16个INT8）为粒度写入，因此融合FSM必须把连续的字节流重新按16个一组打包成tile字；又因为OBUF的读出路径存在组合MUX传播加BRAM寄存读共两拍的流水延迟，状态机专门设置了\texttt{F\_WAIT\_MUX}等待级来吸收这一延迟，保证字节到tile装配时的字节对齐严格正确，从而维持逐位一致。

\section{流水线状态机与TILE\_SPLIT优化}

核心引擎的FSM包含多个状态，其中计算数据通路划分为以下流水线级：

计算流水线（每个input-block迭代）：
\begin{enumerate}
	\item ST\_FETCH / ST\_WAIT\_SRAM：发出BRAM读地址，等待权重SRAM输出（1 cycle latency）；
	\item ST\_XEFF\_REG：输入BRAM输出就绪 + 零点减法 → 锁存x\_eff\_reg；
	\item ST\_MAC（或ST\_MAC\_Q0--Q3）：cim\_tile以时分复用方式计算 x\_eff\_reg $\times$ w\_tile\_reg → 锁存tile\_psum\_reg。当TILE\_SPLIT\_FACTOR=4时，该级按四个切片（Q0--Q3）顺序执行，每个切片含锁存（\_L）、乘积寄存（\_M）与累加子状态，每次处理4列MAC、由4个物理乘法器复用覆盖全部16列；
	\item ST\_COMPUTE：psum\_accum += tile\_psum\_reg（寄存器加法）。
\end{enumerate}

输出流水线（每个output neuron，所有IB迭代完成后）：
\begin{enumerate}
	\setcounter{enumi}{4}
	\item ST\_BIAS\_ADD → ST\_ACTIVATE：加偏置 → ReLU激活；
	\item ST\_STORE：64位乘法（psum $\times$ requant\_mult）→ 锁存prod\_r；
	\item ST\_SHIFT\_CLAMP → ST\_WRITE\_OBUF：移位 + 舍入 + clamp → 写输出缓冲。
\end{enumerate}

上述计算流水线与输出流水线由核心引擎的FSM统一调度，其完整状态转移如图\ref{fig:core_fsm}所示，图中的内层循环（橙色，每IB迭代）完成取数、零点减法与MAC累加；外层循环（蓝色，每output block）完成偏置累加、激活、重量化与结果写回。单一FSM经CSR配置即可服务任意层维度。包含按input-block迭代的内层MAC循环与按output-block迭代的外层循环。

为进一步提升时钟频率，后量化路径（原ST\_SHIFT\_CLAMP）被拆分为ST\_SHIFT（执行rounding加法并锁存pre\_shift\_r）和ST\_CLAMP（barrel shift + clamp至INT8范围）两级流水。该拆分将包含多级CARRY4进位链的组合路径分割为两段，每段关键路径显著缩短。配合Tile内的乘法器时分复用与乘积寄存器流水，本设计在PAR\_OB=2配置下于64 MHz完成时序收敛（WNS +0.013 ns，0个failing endpoints），PAR\_OB=3配置亦在64 MHz收敛（WNS +0.158 ns）；在75 MHz下出现时序违例（PAR\_OB=2时WNS -2.290 ns、72个failing endpoints），在80 MHz下违例更严重（WNS -3.324 ns、2212个failing endpoints），但75/80 MHz配置在室温上板功能完全正确、逐元素结果与golden model一致，作为功能性超频实验保留。

把上述流水级放回整体循环来看，单层的总周期数可以拆解为三部分之和：每个output-block都要把$N_{IB}$个input-block逐个取数并累加，随后对$\text{PAR\_OB}\times\text{TILE\_ROWS}$个输出神经元做一次偏置累加、激活与重量化，如此重复$\lceil N_{OB}/\text{PAR\_OB}\rceil$组，再加上启动、配置与收尾的固定开销。这一可解析的周期模型可与硬件性能计数器读到的实测周期相互印证（各层实测周期见后文表\ref{tab:mlp_perf}与表\ref{tab:lenet5_perf}）。由于MAC段采用时分复用、且偏置/激活/重量化等后处理对每个输出神经元串行执行，单Tile的\emph{有效}MAC吞吐率远低于$16\times16$阵列的峰值256 MAC/cycle，系统级有效吞吐主要由这些串行段而非乘法器峰值决定。提升有效吞吐的可行方向是提高输出并行度\texttt{PAR\_OB}（本设计已借助时分复用在板级部署PAR\_OB=2/3）并对输出/重量化段做并行化或流水化，相关的并行度扫描与瓶颈分析见第4章中关于PAR\_OB扫描的小节。在拆分因子的取舍上，\texttt{TILE\_SPLIT\_FACTOR=4}是一个典型的"以吞吐换频率"决策：它把原本一拍完成的16列MAC摊成四拍并叠加乘积寄存器流水，单看MAC吞吐率确实下降，但换来的是把组合关键路径大幅压缩，使设计能够在64 MHz下闭合时序，同时也为偏置/激活/重量化等非MAC流水级以及双缓冲乒乓的重叠争取到了时序裕量。

综合权衡下，更高的时钟频率与更充分的流水重叠所带来的收益，超过了单纯MAC吞吐率下降的损失。本文统一以64 MHz签核频率作为性能数据的标称频率：计算核延迟由性能计数器实测周期数除以64 MHz折算（周期数本身与频率无关）；端到端延迟为板级实测值，因受DMA带宽主导、已由75 MHz与80 MHz对照实测验证与PL时钟频率基本无关，故该值对工作频率不敏感。75/80 MHz配置存在时序违例但室温上板功能正确、逐元素结果bit-exact，作为功能性超频实验在下文时序分析中单独讨论。

\section{设计动机}

前述ARM PS控制模式下，CIM加速器依赖ARM Cortex-A9进行推理调度与数据搬运。为实现纯PL侧自主推理能力，本设计集成PicoRV32开源RISC-V软核（YosysHQ，MIT License）替代ARM PS。其优势包括：

\begin{itemize}
	\item 单文件实现（\texttt{picorv32.v}），RV32IM指令集；
	\item 资源开销低且反直觉地轻量：PicoRV32核心约1,500个LUT，完整子系统（含bridge、UART TX、FW/Result BRAM接口）因去除了axi\_dma及关联AXI互连，总LUT与FF反而比ARM模式更少（LUT减少336个、FF减少2136个，详见表\ref{tab:util_rv32}和表\ref{tab:rv32_vs_arm}），以更少的逻辑资源实现更自主的控制；
	\item 成熟的native memory interface，适合直接桥接到CIM CSR。
\end{itemize}

最终目标是让CIM加速器在纯PL中自主运行推理，不依赖PS控制。

在软核选型上，本设计在若干主流开源RISC-V核之间做了权衡。VexRiscv虽然性能可观、可裁剪性强，但其核是由SpinalHDL生成的，引入它意味着要额外搭建一套SpinalHDL工具链；SERV是面积极小（约200 LUT）的位串行核，但正因为是串行执行，逐位处理使其运行速度过慢，不适合承担逐字写CSR这类需要一定吞吐的控制任务；Ibex则是工业级、配置完备的RV32IMC核，但相对偏大、配置也更复杂。相比之下，PicoRV32以"单文件Verilog、约1,500 LUT、MIT许可、支持RV32IM（含乘法指令）、且对外暴露极简的native memory interface"这几个特点胜出：单文件意味着可以直接加入Vivado工程而无需任何外部生成器；约1,500 LUT的占用相对xc7z020全片5万余个LUT几乎可以忽略，不会挤占CIM的资源；RV32IM则足以编译运行驱动推理所需的C固件。尤其关键的是其native memory interface——只有valid/ready/addr/wdata/rdata等寥寥数根信号、不涉及AXI或Wishbone的复杂握手，使得后续把它桥接到CIM的CSR地址空间变得相当直接。综合这些因素，PicoRV32是在"足够小、足够成熟、足够好接"三者之间最平衡的选择。

PicoRV32系统与ARM版本共享相同的CIM IP RTL（含TILE\_SPLIT\_FACTOR=4、乘法器时分复用、双缓冲、Layer Fusion、WEIGHT\_BASE/BIAS\_BASE等全部优化），本模式部署PAR\_OB=2配置，同样基于64 MHz签核频率。以下各节给出的性能数据基于64 MHz频率。

\section{系统架构}

PicoRV32集成后的系统顶层架构如图\ref{fig:rv32_soc}所示：PS仅承担固件加载与结果回读，PicoRV32子系统（核心 + bridge + 双端口FW/Result BRAM + AXI GPIO复位控制）在纯PL侧自主驱动推理，右侧CIM IP保持零RTL改动。系统设计要点包括：

（1）CIM IP不做修改：PicoRV32通过mini AXI master发送与ARM完全相同的AXI事务，CIM IP的AXI slave接口无需任何改动，确保了与ARM控制路径的完全兼容。

（2）双端口FW BRAM：固件通过PS侧AXI写入FW BRAM port B，而不是烧写到bitstream中。这使得每次更换测试图片无需重新生成bitstream，只需重新加载firmware.hex。FW BRAM容量32KB，足以容纳784→16→10小型MLP模型的权重数据与C固件代码（共约17.3KB）。

（3）Result BRAM：PicoRV32将推理结果（预测类别、logits、magic标记）写入Result BRAM，PS通过AXI读取结果。这避免了PL侧UART与PS侧USB的连接性问题。

（4）CPU复位控制：PS通过AXI GPIO控制cpu\_rst\_n信号，实现hold → write firmware → release的操作流程。

具体而言，\texttt{picorv32\_cim\_bridge}本身是一个小型状态机。PicoRV32发起访存时拉高\texttt{mem\_valid}并给出地址、写数据与字节选通；bridge在\texttt{S\_IDLE}态对地址高位译码，判断目标是FW BRAM、CIM CSR、UART还是Result BRAM，再分别进入相应的处理分支——访问BRAM类外设走读/写直通，访问CIM则进入等待AXI事务完成的子状态，处理完毕统一回到\texttt{S\_DONE}拉高一拍\texttt{mem\_ready}通知CPU、随后返回\texttt{S\_IDLE}。其中通往CIM的分支尤为关键：由于PicoRV32的native总线不能直接驱动AXI，bridge内嵌了一个mini AXI master，把一次native写/读翻译成标准的AXI4-Lite写（AWVALID+WVALID→等BVALID）或读事务。这样CIM IP的AXI slave接口所看到的与ARM PS发来的事务在时序和语义上完全一致。这一层翻译使得第2章设计的整个CIM IP（接口层之外的计算层与存储层）在更换控制核时能够不修改RTL，最大程度复用已验证的设计。

\section{模型规模约束}

虽然PicoRV32反而节省了LUT资源，但是固件中需要放入32KB FW BRAM，模型规模受到限制(将MLP模型缩小为784→16→10)。FW BRAM中需要存储：C固件代码、权重数据（INT8打包）、偏置数据（INT32）与输入数据。对于784→16→10的小型MLP模型，数据量约为：

\begin{equation}
	784 \times 16 \times 1 + 16 \times 10 \times 1 + (16 + 10) \times 4 + 784 \times 1 \approx 13.3\text{KB}
\end{equation}

加上C固件代码约4KB，总计约17.3KB，可以放入32KB BRAM中。该模型在MNIST测试集上INT8量化推理准确率超过95\%。

另外，关于PicoRV32版本可支持的模型规模还需补充说明，32KB FW BRAM在扣除约4KB固件代码后，约余28KB用于权重、偏置与输入，至多可容纳约28K个INT8参数。例如784→32→10（约25K参数）尚可放入，而784→128→10（约102K参数）或LeNet-5则远超此容量——这正是PicoRV32模式仅验证784→16→10小型MLP、而把较大网络留给ARM模式的根本原因。若要在纯PL侧支持更大模型，需扩大FW BRAM容量，或改由片外存储分批载入权重（代价是引入额外搬运开销）。本项目暂未对此展开深入研究。

\section{PicoRV32子系统设计}

PicoRV32子系统的最终架构采用混合PS+PL设计：通过Vivado Block Design纳入Zynq PS以提供时钟与AXI通道，PicoRV32软核在PL侧自主运行，经由总线桥接模块（\texttt{picorv32\_cim\_bridge}）将native总线事务翻译为标准AXI4-Lite事务以访问CIM CSR、FW BRAM与UART TX。增设双端口Result BRAM：PicoRV32将推理结果写入其一端口，PS通过AXI BRAM Controller从另一端口读出。FW BRAM同样采用双端口结构，PS在运行时通过AXI BRAM Controller写入固件（先拉低\texttt{cpu\_rst\_n}复位CPU，写入完毕后释放复位），使单一bitstream即可覆盖全部测试图片而无需重新综合。

\section{验证方法学}

本项目建立了"Python golden model → VCS仿真 → PYNQ上板"三层一致的验证方法学：

第一层：Python bit-exact参考模型。\texttt{golden\_model.py}以纯整数算术实现与硬件完全相同的计算流程（零点减法、MVM、偏置累加、ReLU、重量化），输出hex格式测试向量供RTL仿真和板级验证使用。

具体而言，该参考模型的核心是一组与RTL流水级一一对应的纯函数：
\newline
\texttt{apply\_zero\_point}完成零点减法、\texttt{cim\_mvm}完成矩阵乘加、\texttt{add\_bias}加偏置、\texttt{relu}做激活、\texttt{requantize\_int32\_to\_int8}做带舍入的定点重定标，再由\texttt{infer\_layer}与\texttt{infer\_mlp}把它们串接为单层与多层推理。这些函数被刻意写成与第2章硬件数据通路的每一级严格对应，并采用与硬件一致的零点处理约定（输入像素先按$[0,255]$无符号取值、零扩展后再减去零点，得到\emph{有符号}有效输入），从而保证软件参考值与硬件输出在每一拍都一致。针对卷积，模型提供了\texttt{im2col\_explicit}与基于地址表的\texttt{im2col\_implicit}两套展开实现，把卷积转化为矩阵乘法后交由同一MVM内核处理，并以\texttt{infer\_conv\_layer}给出卷积层参考输出；\texttt{calibrate\_requant}则用于离线标定每层的requant\_mult与requant\_shift。模型自带的\texttt{self\_test}可在完全脱离硬件的情况下回归整条数值流程，是后续所有bit-exact比对的"真值"来源。

第二层：VCS RTL仿真。采用"单元级→阵列级→系统级"分层验证策略：
\begin{itemize}
	\item tb\_cim\_tile：CIM Tile单元测试，103个测试向量（含3个定向测试与100个随机向量），验证MAC运算正确性；
	\item tb\_cim\_accel\_core：系统级MVM测试，覆盖随机向量与边界用例（全零、全$-128$、最大维度等）；
	\item tb\_mnist\_e2e：MNIST端到端测试，784→128→10两层MLP，与golden model逐元素对比。
\end{itemize}

通过\texttt{run\_regression.sh}脚本一键运行全部测试并汇总PASS/FAIL结果。

这三个testbench覆盖了从原子单元到完整网络的不同验证粒度。\texttt{tb\_cim\_tile}用103个测试向量专门验证16×16阵列的MAC正确性；\texttt{tb\_cim\_accel\_core}则搭建一个IN=32、OUT=PAR\_OB×16（即恰好一组output-block）的小层，通过整行写权重、整tile写输入的接口把数据灌入存储，触发计算后读回输出缓冲并与片内golden逐元素比对——这里有一个容易出错的细节被特别处理：golden计算\,x\_eff\,时必须把输入像素按$[0,255]$的无符号值零扩展后再减去零点（对应RTL中的\texttt{\$signed(\{1'b0, x\_tile\})}），得到\emph{有符号}的x\_eff，而不能把原始8位像素直接按有符号解释，否则当输入大于等于128时软硬件就会分歧；此外该testbench刻意构造了全零、全$-128$、最大维度等边界用例以暴露符号、进位与边界处理上的潜在缺陷。\texttt{tb\_mnist\_e2e}则以784→128→10两层MLP做端到端验证。三者经\texttt{run\_regression.sh}统一驱动，任何一处逐位不一致都会被立即捕获并标红。

第三层：PYNQ-Z2上板验证。将bitstream与hwh文件部署到PYNQ板，通过Python驱动程序（\texttt{cim\_driver.py}的\texttt{CIMModel}类），加载真实MNIST手写数字图片，逐层执行推理并与golden model对比。

为使"硬件与软件逐位一致"这一论断可被逐层审计，板级驱动还内置了逐层比对开关：开启后，驱动会在每层推理完成时把该层的输入、硬件输出、golden输出以及二者的逐元素差异分别dump到约定的日志目录，并对每层断言硬件结果与golden数组完全相等。这样一旦某层出现偏差，便能立即定位到具体的层与具体的元素，而不必从最终分类结果反向猜测出错环节。正是依托这一基础设施，本文后续才得以给出"全部硬件输出与软件参考模型bit-exact一致"的强结论，而非仅以分类准确率间接佐证。

\end{itemize}

两测试集中的错分（测试集一0张、测试集二2张）均与Python golden model的预测完全一致，说明这是INT8量化精度损失导致的，而非硬件计算错误。全部200张图片的全部层输出均与Python参考模型bit-exact一致，验证了硬件实现的正确性。


\section{SQ-mapping：块对角权重打包的卷积映射优化}
\label{sec:sqmapping}

在LeNet-5这类小卷积核网络中，直接把im2col得到的每个输出像素逐一送入MVM引擎会严重浪费CIM阵列的利用率。以Conv1层（$1\times5\times5\to6$）为例，单次MVM的有效输入维度仅为$\text{col\_len} = C_{in}\times K_h\times K_w = 25$，而硬件支持的最大输入维度为1536，利用率不足$25/1536\approx 1.6\%$；同时单次MVM产生$C_{out}=6$个输出神经元，输出侧利用率同样极低。这种"瘦长矩阵挤在大阵列里"的映射方式使得每次硬件调用的建立与DMA开销被完全暴露，无法被有效均摊。

针对这一问题，本文引入并应用了块对角权重打包（SQ-mapping）这一已有的软件映射优化方法：当$\text{col\_len}\times C_{out}$远小于硬件容量时，把$k_{\text{pack}}$个输出像素的im2col列与权重一起打包到同一次MVM调用中。SQ-mapping的基本思想并非本文首创，本文的贡献在于将其与本设计的硬件维度上限（MAX\_IN\_DIM/MAX\_OUT\_DIM）、双缓冲乒乓批处理流程及Python驱动侧的权重预打包相结合，给出在本平台上的具体参数化实现与bit-exact验证。原始权重矩阵$W\in\mathbb{Z}^{C_{out}\times \text{col\_len}}$沿对角线复制$k_{\text{pack}}$份构造块对角矩阵：

\begin{equation}
	W_{\text{packed}} = \begin{bmatrix}
		W & & & \\
		& W & & \\
		& & \ddots & \\
		& & & W
	\end{bmatrix} \in \mathbb{Z}^{(k_{\text{pack}}\cdot C_{out})\times (k_{\text{pack}}\cdot \text{col\_len})}
\end{equation}

输入侧同时将$k_{\text{pack}}$个输出像素对应的im2col列串联成长向量。一次硬件调用即可同时得到$k_{\text{pack}}\cdot C_{out}$个输出神经元，再按$C_{out}$拆回$k_{\text{pack}}$组像素结果。打包因子由硬件维度上限决定：

\begin{equation}
	k_{\text{pack}} = \min\!\left(\left\lfloor\frac{\text{MAX\_IN\_DIM}}{\text{col\_len}}\right\rfloor,\ \left\lfloor\frac{\text{MAX\_OUT\_DIM}}{C_{out}}\right\rfloor\right)
\end{equation}

SQ-mapping的块对角打包思想如图\ref{fig:sqmapping}所示：左侧逐像素映射时瘦长权重矩阵仅占据阵列极小一角，利用率不足5\%且每次调用开销无法均摊；右侧将$k_{\text{pack}}$份权重沿对角线复制构成块对角矩阵，一次MVM调用即可并行处理$k_{\text{pack}}$个输出像素，从而显著削减MVM调用次数。表\ref{tab:sqmapping}给出了LeNet-5两个卷积层在MAX\_IN\_DIM=1536、MAX\_OUT\_DIM=256配置下的打包结果。该优化不需要修改任何RTL，完全在Python驱动侧通过权重矩阵构造与输入向量拼接实现。启用SQ-mapping后，200张LeNet-5图片的逐层bit-exact验证全部通过，分类准确率仍为99.5\%（199/200），与逐像素调用模式完全一致。

\section{加速比分析}

\subsection{硬件理论吞吐率}
\label{sec:hw_throughput}

CIM硬件的核心优势在于利用存储阵列的并行性实现高通量MAC运算。需要区分\emph{标称峰值}与\emph{有效吞吐}：在乘法器时分复用下，单个$16\times16$ Tile每4拍完成一次256-MAC，叠加逐输出神经元的串行偏置/激活/重量化后处理，单层的有效MAC吞吐显著低于阵列的瞬时峰值。以全连接网络MLP（784→128→10）为整体实测（其硬件实际执行约102,400次MAC，含将输出维度对齐至16倍数的padding，分层明细见表\ref{tab:mlp_perf}）：

\begin{itemize}
	\item FC1层（784→128）总MAC操作数：$784 \times 128 = 100{,}352$
	\item FC1层硬件周期数：3,737
	\item 全网MLP实测周期数（PAR\_OB=2）：4,266
	\item 全网有效MAC吞吐：$102{,}400 / 4{,}266 \approx 24.0$ MACs/cycle（含层间切换与后处理串行开销）
	\item 有效吞吐率（64 MHz）：$24.0 \times 64 \times 10^6 \approx 1.536 \text{ GMACs/s}$
\end{itemize}

\noindent 该有效吞吐率远低于$16\times16$阵列的标称峰值，主要瓶颈在于MAC段的时分复用（完整覆盖16列需多拍）与逐神经元串行重量化；提升途径为增大输出并行度\texttt{PAR\_OB}并对输出/重量化段做流水化（详见§\ref{sec:par_ob_sweep}）。

\section{并行度可扩展性的仿真验证}
\label{sec:par_ob_sweep}

\subsection{动机与方法}

如§\ref{sec:two_extremes}所述，本设计的输出并行度由编译期参数PAR\_OB控制：核心通过\texttt{generate}对PAR\_OB份16$\times$16 Tile完全参数化例化。得益于乘法器时分复用（单Tile的DSP48占用由256降至约64），板级实现已可在PYNQ-Z2的220块DSP预算内部署PAR\_OB=2（MAC约128块、实测共134块/61\%）乃至PAR\_OB=3（MAC约192块、实测共198块/90\%）；而PAR\_OB$\geq$4所需的DSP（$\geq$256块）已超出该器件预算，物理上无法在板上例化。这是PYNQ-Z2的器件资源天花板而非架构能力上限——为验证更高并行度下架构的\emph{可综合性}与\emph{功能正确性}，下文通过周期级仿真将PAR\_OB扫描至16。

\subsection{加速比次线性的成因分析}

次线性的根因可由FSM的周期构成精确解释。单个输出分组（ob\_group）的总周期由两部分构成：

（A）乘加阶段（可并行）。权重fetch、x\_eff锁存、4个quarter MAC子状态（TILE\_SPLIT=4）与psum累加，对N\_IB个输入块循环执行。该阶段在所有PAR\_OB份Tile上\emph{并行}完成，因此其总周期随PAR\_OB增大而减小：分组数为N\_OB/PAR\_OB，故乘加总周期从PAR\_OB=1的约2592周期降至PAR\_OB=16的约642周期。

（B）输出/requant阶段（串行）。偏置加法、激活、requant、移位、钳位、写回OBUF构成逐神经元的7级流水（ST\_BIAS\_ADD$\to\cdots\to$ST\_WRITE\_OBUF），对每个输出神经元\emph{逐一串行}处理。该阶段每个分组处理PAR\_OB$\times$16个神经元，而分组数为N\_OB/PAR\_OB，故输出阶段总周期为$\frac{N\_OB}{PAR\_OB}\times PAR\_OB\times 16\times 7 = N\_OB\times 16\times 7$，约1792周期——\emph{与PAR\_OB完全无关，是不可被并行度压缩的固定开销}。

二者叠加即为表\ref{tab:par_ob_sweep}的实测值：随PAR\_OB从1增至16，可并行的乘加阶段占比从约59\%降至约26\%，而固定的输出阶段占比从约41\%升至约74\%，逐渐主导总延迟，遵循Amdahl定律——当固定串行部分约占一半时，无论并行度多高，总加速比上限即约2倍。这一周期模型与实测值吻合至$\pm$2周期（误差仅来自ST\_IDLE/ST\_DONE边沿）。值得强调的是，单纯增大层规模并不能改善该比值：当N\_OB由16增至64时，乘加与输出阶段同比例增长，实测加速比仍恒为约1.80倍。

\subsection{提高并行度收益的途径}

上述分析直接指明了改进方向——\emph{瓶颈不在乘加并行度本身，而在串行的输出/requant流水}。据此可采取以下措施提高加速比：

（1）输出流水线化（最关键）。当前输出阶段每个神经元独占7个时钟周期（逐级走完流水），若将其改造为真正的吞吐流水线——每周期接纳一个神经元、7级填充后每周期输出一个结果——则输出阶段从$N\_OB\times16\times7$降至约$N\_OB\times16\times1$。据周期模型估算，仅此一项即可将PAR\_OB=16的加速比从1.80倍提升至约3.25倍。

（2）权重fetch与计算重叠。当前每个输入块需经ST\_FETCH/ST\_WAIT\_SRAM串行取权重（约$2\times$PAR\_OB周期），可通过预取（在当前块计算时fetch下一块权重）将其隐藏。结合输出流水线化，PAR\_OB=16的加速比可进一步提升至约5.3倍。

（3）输出阶段的并行化。更彻底的方案是让多个Tile的输出共享多套并行的requant单元，使输出阶段也随PAR\_OB缩放；代价是额外的requant逻辑面积，需与DSP/LUT预算权衡。

（4）配合DSP48 INT8 SIMD打包释放并行资源。如§\ref{sec:future_work}所述，利用单DSP48E1的25$\times$18乘法器同时计算两路INT8乘法可将DSP用量再减半（单Tile约32块），从而在同一PYNQ-Z2上将板级并行度由当前的PAR\_OB=2/3进一步提升（如PAR\_OB=4$\sim$6）。需注意：唯有在上述输出流水线化改造之后，提升的并行度才能转化为可观的端到端加速，否则收益将被固定的输出开销吞没。

综上，本节通过周期级仿真确证了阵列架构在PAR\_OB=1$\sim$16下均\emph{可综合且功能完全正确}，板级部署PAR\_OB=2/3（更高并行度仅受DSP资源约束）；同时定量揭示了当前加速比次线性的根因（串行输出流水构成约占半数的固定开销），并给出了从架构层面提高并行度收益的明确路径。

\section{工作总结}

本文设计并实现了一种面向边缘智能推理的INT8数字存算一体矩阵计算阵列架构，在PYNQ-Z2（Zynq-7020）FPGA平台上完成了完整的综合部署与板级验证。主要成果总结如下：

（1）架构设计：以16×16 CIM Tile为原子计算单元，设计了包含多级流水线的核心加速器引擎，支持通过CSR在运行时配置层维度（不超过MAX\_IN\_DIM=1536、MAX\_OUT\_DIM=256）与量化参数。所有设计参数集中管理于cim\_pkg.sv，实现了模块化、参数化的RTL框架。

（2）数据通路优化：设计了AXI4-Stream + DMA双向批量传输架构，将数据搬运从逐字MMIO替换为DMA批量传输；实现了输入/输出缓冲双bank乒乓结构，将DMA传输与硬件计算重叠执行；针对连续FC层设计了OBUF→IBUF内部硬件直拷的层融合机制，消除层间DMA往返。累计将LeNet-5端到端延迟从3,193.52 ms/img降至29.88 ms/img（约107×加速，33.5 fps）。在与ARM Cortex-A9纯软件同口径的系统级端到端对比中，CIM硬件在三种网络上均快于软件（MLP约2.24×、small\_mlp约1.23×、LeNet-5相对batched-BLAS约1.34×），在计算核纯算力口径下更领先约一到两个数量级（约65$\sim$191倍）。

（3）软件映射优化：引入并适配了块对角权重打包（SQ-mapping）这一已有的卷积映射方法，通过将多个输出像素的im2col列合并到同一次MVM调用，将小卷积核场景下的MVM调用次数削减约90\%（Conv1 97.6\%、Conv2 89.1\%），单次MVM维度占用率从不足2\%提升至约60\%--98\%。同时通过im2col/MaxPool向量化、layer-wise批量化等手段将Python侧开销压缩至总延迟的约13\%。

（4）工程问题解决：针对Vivado BRAM推断失败问题，通过16-bank权重SRAM拆分解决；针对关键路径时序违例，通过TILE\_SPLIT\_FACTOR流水拆分与后量化路径两级流水将时序收敛频率提升至64 MHz（WNS +0.013 ns、完全收敛，作为板级签核工作点），并可在75/80 MHz下功能正确运行（时序违例但常温功能正确的超频点）。

（5）软核集成与轻量化：成功集成PicoRV32 RISC-V软核替代ARM PS进行推理控制，CIM IP完全不修改。相比ARM模式，PicoRV32模式展现出显著的轻量化优势：LUT和FF均显著减少（LUT: 15231 vs 15567，-336个；FF: 14654 vs 16790，-2136个），在64 MHz下存在轻微时序违例（WNS -0.160 ns，11个failing endpoints）但常温上板功能正确。资源减少源于不需要DMA IP及其关联AXI Stream与Interconnect逻辑（合计节省超2400个LUT/FF），PicoRV32软核本身仅约1500 LUT且不含复杂流水。200张MNIST图片验证准确率约95.5\%（191/200），全部logits与ARM路径bit-exact一致。

（6）验证完备性：建立了"Python golden model → VCS仿真 → PYNQ上板"三层一致的验证流程。在两个独立测试集上，MLP网络200张图片分类准确率99.5\%、LeNet-5 CNN 200张图片99\%--100\%（二者与golden model均100\% bit-exact一致）。所有优化阶段的硬件输出均与golden model保持100\% bit-exact一致；各配置的分类准确率取决于INT8量化模型本身（见表\ref{tab:arm_multi_model}）。

\section{不足与展望}
\label{sec:future_work}

（1）DSP占用率与并行度：得益于乘法器时分复用，单Tile的DSP48E1占用由256降至约64，板级实现可在PYNQ-Z2的220块DSP内部署PAR\_OB=2（约61\%）乃至PAR\_OB=3（约90\%）；而PAR\_OB$\geq$4所需DSP超出器件预算，未能在板上例化。这是器件资源约束而非架构限制——§\ref{sec:par_ob_sweep}的VCS周期级仿真已证明阵列在PAR\_OB=1$\sim$16下均可综合且功能100\% bit-exact正确。后续可通过DSP48 INT8×2 SIMD打包技术（利用单DSP的25×18乘法器同时计算两个独立INT8乘法，将单Tile DSP用量再减半至约32块）释放资源，将板级并行度由当前的PAR\_OB=2/3进一步提升（如PAR\_OB=4$\sim$6）。但如§\ref{sec:par_ob_sweep}的成因分析所示，单纯提升并行度的收益受串行输出/requant流水（约占半数的固定开销）所限而呈次线性；唯有先将输出阶段流水线化（预期可使PAR\_OB=16的加速比从1.80倍提升至约3.25倍，叠加权重预取后约5.3倍），提升的并行度才能充分转化为端到端加速。二者结合是后续提升硬件compute吞吐的关键路径。

（2）最大维度限制：当前MAX\_IN\_DIM=1536、MAX\_OUT\_DIM=256的配置已使BRAM占用接近PYNQ-Z2的物理上限（ARM模式88.21\%、PicoRV32模式92.50\%），适用于MNIST尺度网络但无法支持CIFAR-10/ImageNet级别模型。后续需通过权重分片加载（超标层分多次加载，消除维度硬上限）或迁移至更大BRAM容量的平台来解决。

（3）更复杂网络与算子支持：当前系统已支持Conv + FC + ReLU + MaxPool组成的网络，但尚未实现AveragePool、BatchNorm、Softmax等操作以及Depthwise Convolution、Residual Connection等结构。BatchNorm可在量化阶段静态折叠入权重与偏置（RTL零改动），MaxPool硬件化可进一步消除当前毫秒量级的Python侧池化开销（见表\ref{tab:latency_breakdown}）。

（4）面向成熟推理框架的集成：当前Python驱动需要为每个网络手动编写im2col与层调度逻辑，扩展到新模型的工程成本较高。未来可将CIM硬件抽象为ONNX Runtime的自定义执行后端，仅需在卷积/全连接算子内部注册硬件调用接口，即可复用成熟推理框架的完整前端图优化与算子调度能力，一次性获得YOLO、MobileNet、SqueezeNet等主流网络的支持。

（5）稀疏计算支持：ReLU激活后的特征图典型稀疏度为40--60\%，当前设计对所有零值元素仍执行完整的MAC运算与数据搬运。通过结构化剪枝结合权重压缩传输、零值跳过FSM，可进一步降低有效计算量与DMA传输量。

（6）设计自动化：当前改变PAR\_OB等编译期参数依赖脚本以sed方式patch cim\_pkg.sv，缺乏优雅性。后续可考虑用Chisel或SpinalHDL等参数化硬件描述语言重构核心加速器，以trait形式声明层级参数，一次elaborate即可出多个配置版本，同时可利用其与Verilator的天然集成简化仿真流程。

总体而言，本文围绕在资源受限的小型FPGA上实现INT8数字存算一体计算阵列的完整验证闭环，完成了从架构设计、数据通路优化、软核集成到三层一致验证的系统性工作；上述不足与展望多集中在并行度、网络规模与算子覆盖的进一步拓展，以及向成熟推理框架和设计自动化方向的演进，它们与本文已落地的工作并不冲突，而是在同一套可复现平台之上自然延伸的下一步。
