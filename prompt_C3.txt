### Decision 1 — Protocol family at the data port

| Candidate | Pros | Cons |
|---|---|---|
| AXI4-Lite (status quo) | trivially simple, no new IP | **proven 99% bottleneck** by A2 profiler |
| AXI4-Full slave on cim_0 (PS bus-master via M_AXI_GP) | no DMA IP needed | M_AXI_GP capped at ~600 MB/s; PS still drives every transaction; software still does word-by-word `MMIO.write` unless you also adopt `pynq.MMIO.array` tricks |
| AXI4-Stream + axi_dma (current draft) | clean software API, Xilinx IP, ~1.2 GB/s via S_AXI_HP | adds ~500 LUT + 2 BRAM; needs CMA buffer; descriptor setup overhead per transfer |
| AXI4-Stream + AXI MCDMA | multi-channel = independent weight/input/bias channels run in parallel | larger area; project may not need true parallel channels |
| AXI4-Stream + AXI CDMA (mem-to-mem) | both endpoints are memory-mapped; could push from a PL-side scratchpad | scratchpad doesn't exist; pointless here |
| Direct connection to S_AXI_ACP (cache-coherent) | avoids cache-flush overhead | ACP is single-port and contended with PS; throughput tops out around 800 MB/s on 7020 |

Recommend one. State the **measured or datasheet** throughput ceiling on PYNQ-Z2 specifically (Zynq-7020 has a single DDR3 controller, ~3.2 GB/s aggregate; HP ports share this). Prove the recommended ceiling exceeds the project's needs by computing the worst-case per-image data volume from `cim_pkg.sv` parameters.

### Decision 2 — Data width

32-bit vs 64-bit on the PS-side AXI? 32-bit vs 64-bit on the AXI Stream into cim_0? The DMA can width-convert internally, but choosing wisely affects:
- BRAM port width on `weight_sram` (currently 128-bit per bank, written one whole row at a time → with 32-bit stream, 4 beats per row; with 64-bit, 2 beats per row).
- LUT cost of `axis_dwidth_converter`.
- DMA setup overhead per transfer (smaller absolute data requires fewer beats; the per-transfer overhead may not amortize for small layers).

Decide and justify with a per-layer beat-count table for MNIST-MLP and LeNet-5.

### Decision 3 — Stream sink internal architecture

Three sub-decisions:

**3a. Single sink with destination CSR vs three separate stream ports.** The current draft uses one stream port + a `CSR_STREAM_DEST` register that must be written before each transfer (extra MMIO round-trip per transfer). Three separate ports eliminate this but add 3× the BD wiring and require either MCDMA or three single-channel DMAs. Recommend.

**3b. Where the 4-beat-to-128-bit row assembly happens.** Inside `cim_axi_stream_sink.sv` (small shift register), or push 128-bit beats end-to-end (requires MM2S width = 128 → 4× DDR efficiency vs port width)? Trade-off: shift-register is trivial RTL but width-converter inside DMA (`c_m_axis_mm2s_tdata_width=128`) is also free.

**3c. tlast handling and length safety.** Should the sink trust `tlast` from DMA (DMA generates tlast on the last beat of `transfer_len`) or independently count beats against `CSR_STREAM_LEN`? Decide what the right interaction is when the two disagree (overflow, underflow, mid-stream reset).

### Decision 4 — CSR layout changes

The current draft proposes adding `CSR_STREAM_DEST/LEN/STATUS` and removing `CSR_WDMA_*` plus `MEM_INPUT_BASE` / `MEM_BIAS_BASE` write windows. Decide:

- Keep both paths active behind a `CSR_CTRL[3]` toggle? (Author's draft suggests yes for A/B testing.) The cost is preserved RTL complexity. Quantify the LUT cost of dual-path retention vs the validation benefit and recommend.
- Should `CSR_STREAM_STATUS` be polling-only or trigger an IRQ on completion? PYNQ's `dma.sendchannel.wait()` already polls the DMA's own status, so a separate cim_0-side IRQ may be redundant. Recommend.

### Decision 5 — Block Design topology

- **Interconnect**: AXI Interconnect (UG059) vs SmartConnect (PG247). The current TCL uses Interconnect via `apply_bd_automation`. SmartConnect on Vivado 2020.2+ is the modern recommendation but consumes more LUT. Decide.
- **Clock domain**: should axi_dma and cim_0 share `FCLK_CLK0` (60 MHz) or run DMA on a faster `FCLK_CLK1` (e.g. 100 MHz) with `axis_clock_converter` between them? The DMA's MM2S throughput is gated by the slower of {DDR ↔ HP port} and {AXIS clock × tdata width}. Derive the actual bottleneck and recommend.
- **Reset**: shared `proc_sys_reset` for both IPs, or separate? A shared reset means a CIM soft-reset also resets the DMA mid-transfer — likely undesirable. Recommend reset domain partitioning.
- **Interrupt**: use `xlconcat` to combine cim_0/`irq_done` and axi_dma/`mm2s_introut` into PS `IRQ_F2P[1:0]`. Specify the bit assignment.

### Decision 6 — Software API

The current driver pre-allocates fixed-size buffers in `__init__`. Alternatives:
- Pre-allocate per-layer buffers at `CIMModel` construction time (avoids reallocation but consumes more CMA up-front).
- Allocate on first use, cache by shape.
- Use `pynq.allocate(shape, dtype, target=overlay.psddr)` explicitly to ensure the buffer is in DDR (not OCM).

Recommend a `CIMDriver` API with explicit lifetime ownership. Specify what happens if the user calls `load_weights()` with a list larger than the pre-allocated buffer (raise vs auto-grow).

Also decide: should the driver expose a `load_weights_async()` that returns a future, enabling overlap of DMA with Python im2col on the next layer? This is the software half of the C2 (double-buffer) story; even without RTL double-buffering, overlapping Python im2col with hardware DMA can reclaim 20–30% of total time per the A2 profiler. Recommend whether this belongs in C3 or in a separate later task.
